# Design: KSP Step Definitions Generator

**Date:** 2026-03-21
**Scope:** kotlin-behave — new `:annotations` and `:ksp` modules
**Status:** Approved

---

## Overview

A KSP annotation processor that reads `.feature` files at compile time and generates a Kotlin
interface with one method per unique step. The user implements the interface — no string matching,
no regex, no `Given("...")` calls. Full compile-time safety and IDE navigation.

---

## Module Structure

```
kotlin-behave/
├── core/
├── kotest/
├── annotations/   ← KMP, zero dependencies, compile-only (SOURCE retention)
└── ksp/           ← JVM-only KSP processor, depends on :annotations + :core
```

**`:annotations`** is a KMP module — it must compile on all targets because user code (which
annotates the implementing class) compiles on all platforms.

**`:ksp`** is JVM-only — KSP processors always run on the JVM at build time regardless of the
target platform.

---

## Annotations

Defined in `:annotations`, applied to the implementing class:

```kotlin
@Target(AnnotationTarget.CLASS)
@Retention(AnnotationRetention.SOURCE)
annotation class BehaveFeature(val path: String)

@Target(AnnotationTarget.CLASS)
@Repeatable
@Retention(AnnotationRetention.SOURCE)
annotation class BehaveType(
    val placeholder: String = "",    // for single-placeholder or DataTable column mode
    val type: KClass<*>,
    val fields: Array<String> = []   // for explicit field-based multi-placeholder mode
)
```

### Three modes of `@BehaveType`

| Mode | Condition | Use |
|---|---|---|
| **Placeholder** | `placeholder` set, `fields` empty | Maps `{placeholder}` in step text or DataTable column to a type |
| **Field auto-detect** | `placeholder` empty, `fields` empty | Finds steps/columns where ALL primary constructor params of `type` appear by name |
| **Field explicit** | `placeholder` empty, `fields` non-empty | Same as auto-detect but column/placeholder names supplied explicitly |

**Errors:**
- All three of `placeholder`, `type`, `fields` set simultaneously → compile error
- `type` has zero primary constructor parameters in auto-detect or explicit mode → compile error
- Two `@BehaveType` entries with the same `placeholder` or overlapping field sets → compile error

### KSP type resolution

At processor time `val type: KClass<*>` is accessed as a `KSType` via
`annotation.arguments`. The processor resolves it to a `KSClassDeclaration` and reads
`primaryConstructor?.parameters` to obtain field names and types. This works for types
defined in the same module or in dependencies (compiled class metadata is always available).
No multi-round processing is required.

---

## Feature File Path Resolution

The processor resolves `.feature` file paths at build time via a KSP processor option:

```kotlin
// In the user's build.gradle.kts:
ksp {
    arg("behave.featureDir", "src/commonTest/resources")  // default value
}
```

Path = `projectDir / behave.featureDir / annotation.path`.
Compile error (not runtime error) if the file is not found.

## Generated Interface

### Example feature file

```gherkin
Feature: Login

  Background:
    Given I have a collection {string}

  Scenario: Successful login
    Given I am on the login page
    When I enter {string} as username
    And I click the {label} button
    Then I see the dashboard

  Scenario Outline: Login with <role>
    Given I am logged in as <role>
    Examples:
      | role  |
      | admin |

  Scenario: Multiple cats
    Given the following cats:
      | name   | color | age |
      | Whisky | black | 3   |

  Scenario: Credentials
    When I enter {username} as {role} with {int} attempts
```

### Annotations on implementing class

```kotlin
@BehaveFeature("features/login.feature")
@BehaveType(placeholder = "label", type = ButtonLabel::class)  // {label} → ButtonLabel
@BehaveType(type = Credentials::class)                         // auto-detect: {username}+{role}
@BehaveType(type = Cat::class)                                 // auto-detect: columns name+color
@BehaveType(placeholder = "age", type = CatAge::class)        // DataTable column "age" → CatAge
class LoginSteps(val compose: ComposeUiTest) : LoginStepsSpec
```

### Generated interface

```kotlin
// Generated by kotlin-behave KSP — do not edit
package com.example.steps

interface LoginStepsSpec {

    // Background
    suspend fun givenIHaveACollection(string: String)

    // Scenario: Successful login
    suspend fun givenIAmOnTheLoginPage()
    suspend fun whenIEnterAsUsername(string: String)
    suspend fun andIClickTheButton(label: ButtonLabel)
    suspend fun thenISeeTheDashboard()

    // Scenario Outline: Login with <role>  (template preserved, not expanded)
    suspend fun givenIAmLoggedInAs(role: String)

    // Scenario: Multiple cats
    suspend fun givenTheFollowingCats(rows: List<GivenTheFollowingCatsRow>)

    // Scenario: Credentials
    suspend fun whenIEnterAsWithAttempts(credentials: Credentials, int: Int)

    companion object {
        // Uses fully qualified name to avoid collision with top-level steps() in :core
        fun steps(factory: () -> LoginStepsSpec): StepDefinitions<LoginStepsSpec> =
            io.mcol.behave.steps.steps(factory) {
                Given("I have a collection \\{string\\}") { (s: String) ->
                    ctx.givenIHaveACollection(s)
                }
                Given("I am on the login page") { ctx.givenIAmOnTheLoginPage() }
                When("I enter \\{string\\} as username") { (s: String) ->
                    ctx.whenIEnterAsUsername(s)
                }
                And("I click the \\{label\\} button") { (label: ButtonLabel) ->
                    ctx.andIClickTheButton(label)
                }
                Then("I see the dashboard") { ctx.thenISeeTheDashboard() }
                Given("I am logged in as {word}") { (role: String) ->
                    ctx.givenIAmLoggedInAs(role)
                }
                Given("the following cats:") { params ->
                    val rows = params.dataTable!!.rows.map { row ->
                        GivenTheFollowingCatsRow(
                            cat = Cat(name = row["name"]!!, color = row["color"]!!),
                            age = CatAge(row["age"]!!)
                        )
                    }
                    ctx.givenTheFollowingCats(rows)
                }
                When("I enter {word} as {word} with {int} attempts") {
                        (username: String, role: String, attempts: Int) ->
                    ctx.whenIEnterAsWithAttempts(Credentials(username, role), attempts)
                }
            }
    }
}

// Generated per-row class for DataTable steps with multiple column groups.
// Name = method name with first letter uppercased + "Row" suffix.
data class GivenTheFollowingCatsRow(
    val cat: Cat,       // columns: name, color (matched by Cat primary constructor params)
    val age: CatAge     // column: age (matched by @BehaveType placeholder="age")
)
```

**Key points:**
- All method calls inside the companion use `ctx.` — `ctx` is the `StepBuilder<C>.ctx` property
  accessible inside the `steps {}` lambda, holding the current scenario's context instance
- Literal step text is regex-escaped by the processor before emitting `Given("...")`
  (e.g. parentheses, dots escaped) so `TypeRegistry.compile()` treats them as literals
- `{placeholder}` tokens in step expressions are kept as-is (TypeRegistry handles them)
- `<variable>` outline tokens are replaced with `{word}` in the emitted step expression

## Method Name Generation Rules

| Step text | Generated method name |
|---|---|
| `Given I am on the login page` | `givenIAmOnTheLoginPage` |
| `When I enter {string} as username` | `whenIEnterAsUsername` |
| `Then I have {int} items` | `thenIHaveItems` |
| `Given I am logged in as <role>` | `givenIAmLoggedInAs` |
| `And I click the {label} button` | `andIClickTheButton` |
| `But I do not see errors` | `butIDoNotSeeErrors` |
| `Given the following cats:` | `givenTheFollowingCats` |

**Rules:**
1. Keyword (`Given`/`When`/`Then`/`And`/`But`) kept as camelCase prefix as-is — `And`/`But`
   are not resolved to their semantic keyword (no inference of preceding keyword)
2. `{placeholder}` tokens stripped from the method name
3. `<variable>` tokens stripped from the method name
4. Remaining words joined in camelCase, trailing colon stripped
5. **Method name collision** (two different step texts produce the same name): numeric suffix
   added — `givenIHaveItems0`, `givenIHaveItems1`
6. **Step text identity** (same normalised text in multiple scenarios): one method, no suffix —
   see Deduplication section

---

## Parameter Generation Rules

| Input token(s) | Parameter type | Parameter name |
|---|---|---|
| `{string}` | `String` | `string` (or `string0`, `string1` for multiples) |
| `{int}` | `Int` | `int` |
| `{long}` | `Long` | `long` |
| `{float}` | `Float` | `float` |
| `{double}` | `Double` | `double` |
| `{word}` | `String` | `word` |
| `<variable>` | `String` | variable name (e.g. `role`) |
| `{label}` + `@BehaveType(placeholder="label", type=ButtonLabel::class)` | `ButtonLabel` | `label` |
| `{name}` `{color}` absorbed by `@BehaveType(type=ButtonLabel::class)` | `ButtonLabel` | `buttonLabel` (camelCase of class name) |
| `{username}` `{role}` absorbed by `Credentials` + remaining `{int}` | `Credentials, Int` | `credentials, int` |
| DataTable + one type covering all columns | `List<T>` | `rows` |
| DataTable + multiple column groups | `List<GeneratedRow>` | `rows` |
| DataTable + no type registered | `List<DataTableRow>` | `rows` |

## Deduplication

**Step text identity** and **method name collision** are independent concerns:

- **Step text identity:** Same normalised step text in multiple scenarios → one method.
  Normalisation: lowercase, replace all `{placeholder}` and `<variable>` with `{}`, trim.
  If two steps normalise identically but have different placeholder types → first occurrence
  wins, compiler warning emitted.

- **Method name collision:** Two different step texts that produce the same camelCase name →
  numeric suffix added to both: `givenIHaveItems0`, `givenIHaveItems1`.

---

## Unified Parameter Grouping (Inline + DataTable)

The same `@BehaveType` resolution algorithm applies to **both** inline step parameters and
DataTable columns:

- **Inline params** → grouped fields collapse into typed objects → **multiple method parameters**
- **DataTable columns** → grouped fields collapse into typed objects → **`List<GeneratedRow>`**

### Resolution priority (applied in order)

1. Field explicit types (`fields` non-empty) — absorb the named placeholders/columns into `T`
2. Field auto-detect types (no `placeholder`, no `fields`) — absorb all matching by field name
3. Placeholder types (`placeholder` set) — absorb the single named placeholder/column
4. Remaining tokens — kept as their native built-in type (`String`, `Int`, etc.)

**Overlap error:** If two `@BehaveType` entries claim the same placeholder or column name
(e.g. `TypeA(name, color)` and `TypeB(name, age)` both claim `name`) → compile error,
require explicit `fields` on at least one to resolve the ambiguity.

### Multiple groups in one step

```
"I enter {username} as {role} with {int} attempts"
+ @BehaveType(type=Credentials::class)  → absorbs {username} + {role}
+ {int} remains as Int
```

Generated:
```kotlin
suspend fun whenIEnterAsWithAttempts(credentials: Credentials, int: Int)
```

## Scenario Outline Handling

The KSP processor uses its **own lightweight parsing pass** on the raw feature file text —
it does NOT use the runtime `GherkinParser` (which expands outlines before the processor
can see the template). The processor reads outline step templates with `<variable>` tokens
intact before expansion.

When emitting the step expression in the companion:
- `<variable>` in step text → replaced with `{word}` (matches any single word at runtime)
- Variable name used as parameter name in the method signature

---

## Generated Row Class

For DataTable steps with multiple column groups, one data class is generated per step:

**Naming:** method name with first letter uppercased + `Row` suffix.
`givenTheFollowingCats` → `GivenTheFollowingCatsRow`

**Location:** same generated file as the interface, same package as the annotated class.

**Single group shortcut:** If only one `@BehaveType` entry covers all DataTable columns,
no Row class is generated — the method receives `List<T>` directly.

## KSP Processor Pipeline

```
1. Find all classes annotated with @BehaveFeature
2. For each class:
   a. Read @BehaveType annotations:
      - Resolve KSType → KSClassDeclaration → primaryConstructor.parameters
      - Build type override map: placeholder/fields → (type, constructor params)
      - Validate: no overlaps, no zero-field types, no all-three-params conflict
   b. Resolve feature file: projectDir / behave.featureDir / annotation.path
      → compile error if not found
   c. Parse raw feature file (own lightweight pass preserving outline templates)
   d. Extract unique steps: Background + all Scenarios + Outline templates (pre-expansion)
   e. Deduplicate by normalised step text
   f. For each unique step:
      - Escape literal text (regex special chars outside placeholders)
      - Resolve inline parameter groups via @BehaveType algorithm
      - Resolve DataTable column groups if step has a table
      - Generate method name (camelCase, collision suffix if needed)
      - Generate parameter list
   g. Generate Row data classes for multi-group DataTable steps
   h. Generate interface + companion object steps()
      - All method calls prefixed with ctx.
      - Fully qualified io.mcol.behave.steps.steps() to avoid import collision
   i. Write to build/generated/ksp/<target>/kotlin/<package>/
      - Package = same as annotated class
```

## Usage

```kotlin
// 1. Annotate your implementing class
@BehaveFeature("features/login.feature")
@BehaveType(type = Cat::class)
@BehaveType(placeholder = "age", type = CatAge::class)
class LoginSteps(val compose: ComposeUiTest) : LoginStepsSpec {

    override suspend fun givenIHaveACollection(string: String) {
        ctx.compose.collectionsScreenRobot { createCollection(string) }
    }

    // ... implement remaining methods
}

// 2. Wire to runner
gherkin("features/login.feature", LoginStepsSpec.steps { LoginSteps(compose) })
```

## Error Handling (Compile-Time)

| Situation | Outcome |
|---|---|
| Feature file not found | Compile error with full resolved path |
| `@BehaveType` with placeholder + fields + type all set | Compile error |
| `@BehaveType` auto-detect on type with zero constructor params | Compile error |
| Two `@BehaveType` entries claim same placeholder or overlapping fields | Compile error |
| `@BehaveType` field auto-detect matches no step | Compile warning, fallback to String params |
| `@BehaveType` field auto-detect matches multiple steps | Compile error, require explicit `fields` |
| Two steps normalise to same method name | Numeric suffix, no error |
| Two steps normalise to same step text with different param types | Compile warning, first wins |
| DataTable column in `@BehaveType` not found in table | Compile warning |

## Testing Strategy

- Unit tests for the processor using `com.github.tschuchortdev:kotlin-compile-testing-ksp`
  (the standard KSP processor test harness)
- Test feature files covering: Background, Scenario, Outline, DataTable, all placeholder types
- Tests for each `@BehaveType` mode: placeholder, field auto-detect, explicit, DataTable column
- Tests for unified grouping: inline multi-param, DataTable multi-group, mixed
- Tests for deduplication, name collision suffixes, Row class generation
- Tests for all compile-error and compile-warning conditions
- Integration test: annotated class compiles → interface is correct → `steps()` runs against
  real GherkinRunner with a `.feature` file

## Future Extension: `@BehaveTable` (DataTable-Only Shorthand)

When a class is used exclusively for DataTable row mapping and all column names match field names,
a dedicated `@BehaveTable` annotation could replace `@BehaveType`:

```kotlin
@BehaveTable
data class Cat(val name: String, val color: String)
```

Separate spec, plan, and implementation cycle.
