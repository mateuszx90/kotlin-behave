# KSP Code Generation

[‚Üê Back to README](../README.md)

The `:ksp` processor reads your `.feature` files at **build time** and generates a
type-safe `*StepsSpec` interface ‚Äî one method per unique step. You implement the interface;
the compiler tells you if a step is missing. No runtime string matching, full IDE navigation.

## Setup

```kotlin
// build.gradle.kts of the module containing your step definitions
plugins {
    id("com.google.devtools.ksp") version "2.1.20-1.0.32"
}

dependencies {
    testImplementation("io.mcol.kotlin-behave:core:0.1.0")
    testImplementation("io.mcol.kotlin-behave:kotest:0.1.0")
    testImplementation("io.kotest:kotest-runner-junit5:5.9.1")   // JVM
    compileOnly("io.mcol.kotlin-behave:annotations:0.1.0")
    kspTest("io.mcol.kotlin-behave:ksp:0.1.0")
}

ksp {
    // Path relative to the project root where .feature files live.
    arg("behave.featureDir", "src/test/resources")     // default: src/commonTest/resources
    arg("behave.projectDir", projectDir.absolutePath)
}

tasks.withType<Test> { useJUnitPlatform() }
```

> **KMP note:** for a multiplatform module use the matching source-set configurations,
> e.g. `kspJvmTest(...)` / `kspDesktopTest(...)` and put features under the test source set
> referenced by `behave.featureDir`.

Annotate your class ‚Äî KSP generates the spec, the wiring `val`, and the Kotest test:

```kotlin
@BehaveFeature("features/login.feature")
class LoginSteps(val compose: ComposeUiTest) : LoginStepsSpec {
    override suspend fun givenIAmOnTheLoginPage() {
        compose.onNodeWithTag("login_screen").assertIsDisplayed()
    }
    override suspend fun whenIEnterAsUsername(string: String) {
        compose.onNodeWithTag("username_field").performTextInput(string)
    }
    // ... implement every generated method
}
```

`@BehaveFeature(path, generateTest = true)`:
- **`path`** ‚Äî the `.feature` file, relative to `behave.featureDir`.
- **`generateTest`** ‚Äî when `true` (default), KSP also generates `val generatedLoginSteps`
  and a Kotest `FreeSpec` class. Set `false` when you need fully manual wiring (custom
  `parameterType()` registration, tag filtering); then call `gherkin(...)` yourself.

## The generated interface

Given `features/login.feature`:

```gherkin
Feature: Login

  Background:
    Given I have a collection {string}

  Scenario: Successful login
    Given I am on the login page
    When I enter {string} as username
    Then I see the dashboard

  Scenario Outline: Login as <role>
    Given I am logged in as <role>
    Examples:
      | role  |
      | admin |

  Scenario: Multiple cats
    Given the following cats:
      | name   | color | age |
      | Whisky | black | 3   |
```

KSP generates:

```kotlin
// Generated by kotlin-behave KSP ‚Äî do not edit
interface LoginStepsSpec {
    suspend fun givenIHaveACollection(string: String)
    suspend fun givenIAmOnTheLoginPage()
    suspend fun whenIEnterAsUsername(string: String)
    suspend fun thenISeeTheDashboard()
    suspend fun givenIAmLoggedInAs(role: String)                  // <role> ‚Üí String
    suspend fun givenTheFollowingCats(rows: List<GivenTheFollowingCatsRow>)

    companion object {
        fun steps(factory: () -> LoginStepsSpec): StepDefinitions<LoginStepsSpec> = /* ... */
    }
}

data class GivenTheFollowingCatsRow(val name: String, val color: String, val age: String)
```

## Method-name generation

| Step text | Generated method name |
|---|---|
| `Given I am on the login page` | `givenIAmOnTheLoginPage` |
| `When I enter {string} as username` | `whenIEnterAsUsername` |
| `Then I have {int} items` | `thenIHaveItems` |
| `Given I am logged in as <role>` | `givenIAmLoggedInAs` |
| `And I click the {label} button` | `andIClickTheButton` |
| `Given the following cats:` | `givenTheFollowingCats` |

Two steps that produce the same name get numeric suffixes: `thenIHaveItems0`, `thenIHaveItems1`.

## `@Type` + `@TypeConverter`

`@Type` turns a step parameter into a **domain type** instead of a raw `String` ‚Äî applied
directly on the parameter of your step override, with no class-level annotation. A
`@TypeConverter` function in the **same compilation unit** does the mapping.

### One token ‚Üí enum

```gherkin
When Drop item type "square"
```

```kotlin
enum class Item { SQUARE, LINE, T, L /* ... */ }

@TypeConverter
fun toItem(name: String) = Item.valueOf(name.uppercase())     // {string} token ‚Üí enum

@BehaveFeature("features/board.feature")
class BoardSteps : BoardStepsSpec {
    override suspend fun dropItemType(@Type(Item::class) item: Item) {
        board.drop(item)                                       // "square" ‚Üí Item.SQUARE
    }
}
```

### Many tokens ‚Üí one type

A converter can take **as many parameters as the step has tokens** ‚Äî they're matched left to
right and bound into a single typed argument. Here two `{int}` tokens become one `Size`:

```gherkin
Given I have board size 8 10
```

```kotlin
data class Size(val width: Int, val height: Int)

@TypeConverter
fun convertToSize(width: Int, height: Int) = Size(width, height)   // two {int} tokens ‚Üí Size

@BehaveFeature("features/board.feature")
class BoardSteps : BoardStepsSpec {
    override suspend fun iHaveBoardSize(@Type(Size::class) size: Size) {
        board = Board(size.width, size.height)                     // 8 10 ‚Üí Size(8, 10)
    }
}
```

Both function shapes are accepted ‚Äî pick whichever reads better:

```kotlin
@TypeConverter fun convertToSize(width: Int, height: Int) = Size(width, height)   // plain function
@TypeConverter fun String.toItem(): Item = Item.valueOf(uppercase())             // extension function
```

> The generator invokes the converter by its fully-qualified name with the tokens as
> positional arguments, so extension and non-extension converters generate identical, correct
> calls.

**`@Type` vs `@BehaveType`:** use `@Type` on a parameter for a quick "these token(s) are a
domain type" mapping driven by a `@TypeConverter`. Use `@BehaveType` (below) at the class
level when one type should absorb tokens or DataTable columns **by matching their names**.

## `@BehaveType` ‚Äî custom placeholder / DataTable types

Class-level, repeatable. Overrides how `{placeholder}` tokens or DataTable columns map to types.

### Placeholder mode ‚Äî one token ‚Üí one type

```kotlin
@BehaveFeature("features/login.feature")
@BehaveType(placeholder = "label", type = ButtonLabel::class)
class LoginSteps : LoginStepsSpec
```

`And I click the {label} button` ‚Üí `suspend fun andIClickTheButton(label: ButtonLabel)`.

### Field auto-detect mode ‚Äî one type absorbs matching tokens

Absorbs every token whose name matches a primary-constructor parameter of the type:

```kotlin
data class Credentials(val username: String, val role: String)

@BehaveFeature("features/login.feature")
@BehaveType(type = Credentials::class)          // absorbs {username} + {role}
class LoginSteps : LoginStepsSpec
```

`When I enter {username} as {role} with {int} attempts`
‚Üí `suspend fun whenIEnterAsWithAttempts(credentials: Credentials, int: Int)`.

### Field explicit mode

```kotlin
@BehaveType(type = Credentials::class, fields = ["username", "role"])
```

### DataTable mode

Works identically for DataTable columns ‚Äî compose multiple types across one row:

```kotlin
data class Cat(val name: String, val color: String)

@BehaveFeature("features/cats.feature")
@BehaveType(type = Cat::class)                          // columns name + color
@BehaveType(placeholder = "age", type = CatAge::class)  // column age ‚Üí CatAge
class CatSteps : CatStepsSpec
```

```kotlin
data class GivenTheFollowingCatsRow(val cat: Cat, val age: CatAge)
suspend fun givenTheFollowingCats(rows: List<GivenTheFollowingCatsRow>)
```

## `@BehaveCast` ‚Äî lossy numeric casts

When a step uses `{int}` but some Examples values are decimals (e.g. `5.5`), annotate the
parameter with `@BehaveCast` to suppress the type error and generate a truncating conversion:

```kotlin
override suspend fun thenIHaveItems(@BehaveCast int: Int) { /* 5.5 ‚Üí 5 */ }
```

## Sharing steps across features

When the same Gherkin step text appears in 2+ `.feature` files, KSP **errors** unless you opt
in ‚Äî silent duplication is never allowed.

- **`@StepsMixin`** ‚Äî declare an interface with default-bodied step methods; any generated
  `*StepsSpec` whose signature matches auto-extends it, so the body is written once.
- **`@DivergentStep`** ‚Äî mark **every** override when the same step needs intentionally
  different bodies per feature.

See the [README section](../README.md#sharing-steps-across-features) for full examples, and
runnable [example 18](../examples/src/test/kotlin/io/mcol/behave/examples/ex18_shared_mixin/)
/ [example 19](../examples/src/test/kotlin/io/mcol/behave/examples/ex19_divergent_steps/).

## Compile-time errors and warnings

| Situation | Outcome |
|---|---|
| Feature file not found | **Error** with full resolved path |
| `@BehaveType` auto-detect on a type with zero constructor params | **Error** |
| Two `@BehaveType` entries claim the same placeholder or overlapping fields | **Error** |
| Same step text in 2+ features, not covered by `@StepsMixin` / `@DivergentStep` | **Error** naming every class |
| `@BehaveType` field auto-detect matches no step | **Warning**, fallback to `String` |
| Two steps normalise to the same text with different param types | **Warning**, first wins |
| Two steps produce the same method name | Numeric suffix added, no error |
