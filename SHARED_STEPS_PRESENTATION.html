@SharedSteps Architecture

Eliminating Step Definition Duplication in kotlin-behave

The Problem: Step Duplication

Your App Has 3 Feature Tests:
  • recipes-list.feature
  • settings.feature
  • checkout.feature
All 3 Start With The SAME Steps:
  • Given I skip the onboarding
  • When I login
  • And I accept permissions

Generated Interfaces (WITHOUT Fix)

RecipesStepsSpec: ✗ fun skipOnboarding() (REPEATED) ✗ fun login() (REPEATED) ✗ fun acceptPermissions() (REPEATED) + fun viewRecipeList() + fun selectRecipe() SettingsStepsSpec: ✗ fun skipOnboarding() (COPY!) ✗ fun login() (COPY!) ✗ fun acceptPermissions() (COPY!) + fun openSettings() + fun changeLang() CheckoutStepsSpec: ✗ fun skipOnboarding() (COPY!) ✗ fun login() (COPY!) ✗ fun acceptPermissions() (COPY!) + fun addToCart() + fun checkout()
6
Methods repeated
3x
Times copied
300+
Lines of duplicate code
Issues:
  • If you fix a bug in login(), must update 3 places
  • Easy to miss updating one copy
  • Violates DRY principle
  • Scales poorly (add 10 more features → 30x duplication)

The Solution: @SharedSteps Annotation

Step 1: Mark Shared Feature

@SharedSteps("common/onboarding-flow.feature")
class OnboardingSteps : OnboardingStepsSpec {
    override suspend fun skipOnboarding() { ... }
    override suspend fun login() { ... }
    override suspend fun acceptPermissions() { ... }
}

👆 Implement ONCE

Step 2: Declare Usage in Feature Tests

@BehaveFeature(
    "features/recipes.feature",
    uses = [OnboardingSteps::class]  👈 NEW!
)
class RecipesSteps(
    onboarding: OnboardingSteps = OnboardingSteps()
) : RecipesStepsSpec,
    OnboardingStepsSpec by onboarding {  👈 DELEGATE
    override suspend fun viewRecipeList() { ... }
    override suspend fun selectRecipe() { ... }
}

Step 3: KSP Generates With Inheritance

OnboardingStepsSpec
fun skipOnboarding()
fun login()
fun acceptPermissions()
⬇️ INHERITED
RecipesStepsSpec
fun viewRecipeList()
fun selectRecipe()
(3 inherited above)
Result:
  • ✓ One interface per feature
  • ✓ Shared methods inherited, not repeated
  • ✓ Zero generated duplication

Runtime: How Tests Execute Shared Steps

Your Test Code

class RecipesGherkinTest : FreeSpec({
    gherkin(
        "features/recipes.feature",
        RecipesStepsSpec.steps { RecipesSteps() }
                         ^^^^^^ auto-wired!
    )
})

What Gets Registered at Runtime

RecipesStepsSpec.steps() generates StepDefinitions: ┌─────────────────────────────────────────┐ │ Recipe-Specific Steps (from RecipesSteps) │ ├─────────────────────────────────────────┤ │ When "I view the recipe list" → viewRecipeList │ And "I select recipe" → selectRecipe └─────────────────────────────────────────┘ ┌─────────────────────────────────────────┐ │ Shared Steps (auto-merged) │ ├─────────────────────────────────────────┤ │ Given "I skip the onboarding" → skipOnboarding │ When "I login" → login │ And "I accept permissions" → acceptPermissions └─────────────────────────────────────────┘

Feature File Execution

Feature: Recipe List
  Scenario: Browse recipes
    Given I skip the onboarding         ✓ OnboardingSteps
    When I login                        ✓ OnboardingSteps
    And I accept permissions            ✓ OnboardingSteps
    When I view the recipe list         ✓ RecipesSteps
    And I select recipe "Pizza"         ✓ RecipesSteps
Execution Result:
  • ✓ Shared steps executed from shared implementation
  • ✓ Feature steps executed from feature implementation
  • ✓ No duplication, clean separation of concerns

Benefits: Why This Matters

1️⃣ Zero Code Duplication

Before

3 features × 3 shared methods = 9 lines of duplicate code

After

3 shared methods implemented 1× = 3 lines of code

67% less code

2️⃣ Bug Fixes Are Easy

One of your tests fails on login()?

Without @SharedSteps

  • Fix in RecipesSteps.login()
  • Fix in SettingsSteps.login()
  • Fix in CheckoutSteps.login()
  • Pray you didn't miss one

With @SharedSteps

  • Fix in OnboardingSteps.login() — DONE
  • All tests use the same implementation

3️⃣ Type-Safe Inheritance

The compiler knows:

  • RecipesStepsSpec extends OnboardingStepsSpec
  • All required methods implemented

Missing implementation = compile error

4️⃣ Scales Beautifully

Add 10 more features with onboarding?

Without @SharedSteps

  • Copy-paste steps 10 times
  • 100+ lines of code
  • 10 places to update if bug found

With @SharedSteps

  • uses = [OnboardingSteps::class]
  • 1 line per feature
  • 1 place to fix bugs

5️⃣ Clear Architecture
